Two years ago I wrote a post on [DayZ server browsers and the A2S protocol](/2024/05/23/dayz-server-browsers/) (you should read that first).  At the end of the post I mentioned the `A2S_RULES` query and a follow up post to cover how DayZ packs data into that response.  Well, it's much later than I intended, but here it is!

To begin with, we will extend our existing Go code for sending Steam Server Queries.  Instead of sending `A2S_INFO` this time, we will instead send `A2S_RULES`, which is `0x56`. This is, again, guarded by the challenge system, so that code stays in place.  Our example server this time is the [WILDLANDZ](https://wildlandz.com) Green County server at `51.81.8.81:27017`.

```golang
package main

import (
	"bytes"
	"fmt"
	"net"
	"os"
)

func main() {
	var rulesPacket bytes.Buffer

	// packet type
	rulesPacket.Write([]byte{0xFF, 0xFF, 0xFF, 0xFF})
	// request type
	rulesPacket.WriteByte(0x56)
	// payload (0xFFFFFFFF to get challenge)
	rulesPacket.Write([]byte{0xFF, 0xFF, 0xFF, 0xFF})

	conn, err := net.Dial("udp", "51.81.8.81:27017")
	if err != nil {
		panic(err)
	}
	defer conn.Close()

	response := sendPacket(conn, rulesPacket.Bytes())

	if response[4] == 0x41 {
		fmt.Println("Challenge packet received")
		challenge := response[5:]
		fmt.Printf("Challenge: %v\n", challenge)

		// clear challenge request bytes
		rulesPacket.Truncate(5)
		// add the challenge bytes
		rulesPacket.Write(challenge)

		response = sendPacket(conn, rulesPacket.Bytes())
	} else {
		fmt.Println("Challenge packet not received")
	}

	fmt.Printf("Response: %v\n", response)
}

func sendPacket(conn net.Conn, pkt []byte) []byte {
	_, err := conn.Write(pkt)
	if err != nil {
		panic(err)
	}

	response := make([]byte, 1024)

	n, err := conn.Read(response)
	if err != nil {
		panic(err)
	}

	return response[:n]
}
```

# Metadata Smuggling

Turns out, Bohemia is using the `A2S_RULES` response...creatively.  There is a second data structure which is then packed into the response in chunks.  Once the chunks are reassembled they can be parsed as their own data structure. This is something that is only _very_ lightly documented, but has been used with other games.  The main information I can find on it is on their wiki and specific to Arma 3; [Arma 3: ServerBrowserProtocol2](https://community.bistudio.com/wiki/Arma_3:_ServerBrowserProtocol2) and [Arma 3: ServerBrowserProtocol3](https://community.bistudio.com/wiki/Arma_3:_ServerBrowserProtocol3)

Based on those documents, each chunk has a metadata header which contains two uint8 that are the number of chunks and which chunk this is (the index).  Then, the value has binary data packed in as a null-terminated string. The largest size for these chunks appears to be 128 bytes, or 127 bytes of data plus the null termination.

One last complication is escape sequences.  The docs show that there are three escape sequences, though it is not clear why. Obviously the null byte can not be sent as data, but it is not included in an escape sequence. This is quite probably a legacy issue and it doesn't bear thinking about too much.

|Escaped|Unescaped|
|-------|---------|
|`0x01 0x01` | `0x01` |
|`0x01 0x02` | `0x02` |
|`0x01 0x03` | `0x03` |

## Mods

After this comes our first mod information block. This _does_ match up to the documents above.  We get a 4-byte hash of the mod (how to generate this on the client is unknown), then we have length encoded values for the Steam ID and the mod name.  This is important for the Steam ID to conserve space, and the mod name because we can't include any null bytes, so the usual C strings won't work here.

The Steam ID length is number of bytes, so a `1` means read a `uint8`, a `2` means read a `uint16` and a `3` or `4` means read a `uint32`.  I am unsure if a value of `5` and use of a `uint64` is possible, but the mod Steam ID's are still far away from the max `uint32`.  The most recent DayZ mod on Steam at the time of writing this was `3_760_884_271`, which leaves room for about 534 million more mods.
